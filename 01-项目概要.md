# 在线产品目录（Online Product Catalog）— 项目概要

> 版本：v0.1（初稿）
> 日期：2026-07-01
> 状态：立项准备阶段

---

## 1. 项目背景与目标

为门店提供一个**手机端的在线产品展示目录**，让客户在手机上浏览产品、查看价格，并把想要的产品加入购物车、提交订单。

关键定位：

- **不做在线支付**。系统只负责收集"客户想买什么"，不处理收款。
- 订单提交后，进入后台（backoffice）由门店人员查看，**再到我们自己的销售系统（EZ POS）中手工开单**并跟进后续处理。
- 换句话说，本系统是销售系统的**前置获客与下单入口**，而不是完整的电商交易闭环。

### 目标

1. 客户端：随时随地用手机浏览产品、看到"对他适用的价格"、下单。
2. 后台端：门店人员能及时看到新订单、管理产品与价格、处理订单状态。
3. 后端服务：稳定地提供产品/价格/订单数据接口，未来可与 EZ POS 打通。

---

## 2. 系统组成

项目由 4 个独立子项目组成，位于同一根目录下：

| 子项目 | 角色 | 技术栈 | 说明 |
|--------|------|--------|------|
| `catalog-service` | 后端服务（API） | Spring Boot 4.1.0 · Java 25 · Maven · PostgreSQL · Spring Data JPA · Spring Security · Validation · Web MVC | 提供产品、价格、购物车、订单等 REST 接口；数据持久化 |
| `catalog-web` | 客户端手机网站 | **Next.js 16** · React 19 · **antd-mobile** · TypeScript 5 · pnpm | 面向门店客户的移动端产品展示与下单；**保留 Next.js 用于 SEO/SSR** |
| `catalog-backoffice-web` | 后台管理 Web | **Vite + React 19（SPA）** · **antd** · TypeScript 5 · pnpm | 面向门店员工：管理产品/价格、查看与处理订单；**内部工具，去掉 Next.js** |
| `catalog-docs` | 文档 | Markdown | 开发文档、设计决策、接口约定等 |

> 注意：`catalog-web` 与 `catalog-backoffice-web` 目前都是 `create-next-app` 的初始状态；`catalog-service` 仅有启动类；`catalog-docs` 为空。以下内容为**规划蓝图**，用于指导后续开发。
>
> **前端框架决策（2026-07-01）**：
> - **catalog-web 保留 Next.js + antd-mobile**：客户端需要 SEO（产品被搜索/分享），Next.js 提供服务端渲染与每产品页 meta 标签；antd-mobile 官方支持 SSR（见官方 SSR 指南与 `antd-mobile-samples/web-ssr` 示例）。
> - **catalog-backoffice-web 改为 Vite + React SPA + antd**：后台是登录后内部工具，无 SEO 需求，Next.js 的 SSR/RSC 只增加摩擦；antd 为重客户端库，纯 SPA 配合最顺。此项目需从 Next.js 迁移到 Vite（当前仍是 create-next-app 初始态，迁移成本低）。

## 3. 核心功能

### 3.1 客户端（catalog-web）

- **产品浏览**：分类 / 列表 / 搜索，产品详情（图片、描述、规格）。
- **价格展示**：根据客户所属级别显示对应价格（见第 4 节多级价格）。
- **购物车**：加入 / 修改数量 / 删除 / 查看小计。
- **提交订单**：填写基本信息（联系人、备注等）后提交，**无需在线支付**。
- **订单确认**：提交后给出订单号 / 提示，方便客户与门店对接。

### 3.2 后台端（catalog-backoffice-web）

- **订单管理**：查看新订单、订单详情、更新订单状态（如：待处理 / 已开单 / 已完成 / 已取消）。
- **产品管理**：新增 / 编辑 / 上下架产品、图片、分类。
- **价格管理**：维护 list price 及各级别价格。
- **（可选）客户/级别管理**：维护客户及其价格级别。
- **登录鉴权**：员工账号登录，仅授权人员可访问。

### 3.3 后端服务（catalog-service）

- 提供上述所有能力的 REST API。
- 数据校验（Validation）、鉴权（Spring Security）。
- 数据持久化到 PostgreSQL。

## 4. 多级价格模型（核心业务规则）

产品价格分多档：

- **List Price（标准/零售价）** — 基准价格。
- **额外 3–4 级价格** — 例如面向不同客户等级（VIP、批发一级、批发二级……）的价格。

需要明确的设计点（见第 7 节"待确认"）：

1. **谁能看到哪一档价格？** 例如：未登录客户只看 list price；登录后按其账户级别看对应价；或用访问码区分。
2. **价格如何录入？** 每个产品对每一档单独定价，还是基于 list price 的折扣率（如 9 折、8.5 折）自动计算？
3. **级别数量是否固定？** 建议先设计成可配置（1 个 list + N 个级别），而不是写死 4 档。

### 建议的数据模型（初稿）

```
Product        产品：id, 名称, 描述, 分类, 图片, 状态(上/下架), list_price ...
PriceLevel     价格级别：id, 名称(如"VIP/批发一级"), 排序
ProductPrice   产品-级别价格：product_id, price_level_id, price   (决定某产品在某级别的价格)
Customer       客户：id, 名称, 联系方式, price_level_id (客户所属级别)  [是否需要视登录方案而定]
Cart / CartItem 购物车（可放前端本地，也可持久化）
Order          订单：id, 订单号, 客户信息, 状态, 合计, 创建时间, 备注
OrderItem      订单明细：order_id, product_id, 数量, 成交单价(下单时快照)
```

> 关键点：**订单明细必须保存下单时的成交单价快照**，避免日后改价影响历史订单。

## 5. 关键业务流程

### 5.1 下单流程

1. 客户在 `catalog-web` 浏览产品 → 看到适用价格。
2. 加入购物车 → 调整数量。
3. 提交订单（填联系人 + 备注，无支付）。
4. `catalog-service` 生成订单，状态 = **待处理**。
5. 门店员工在 `catalog-backoffice-web` 看到新订单。
6. 员工在 **EZ POS** 中手工开单处理，回到后台把订单状态更新为 **已开单/已完成**。

### 5.2 产品与价格维护流程

1. 员工在后台新增/编辑产品，上传图片、设分类。
2. 维护 list price 及各级别价格。
3. 上架后即在客户端可见。

## 6. 非功能性需求（初步）

- **移动端优先**：客户端针对手机屏幕设计，加载快、操作简单。
- **权限**：后台必须登录鉴权；客户端价格可见性按方案控制。
- **数据一致性**：订单金额、明细一经提交即快照固定。
- **可扩展**：预留未来与 EZ POS 的接口打通（目前人工）。
- **中文优先**：界面与内容以中文为主（按门店实际需要）。

---

## 7. 待确认的关键决策

> 这些问题会直接影响数据模型与开发顺序，建议尽早拍板。

1. **客户端价格可见性方案**：未登录只看零售价？还是必须登录/输入访问码才能看批发价？还是全部公开？
2. **是否需要客户账户体系**：客户要不要注册/登录？还是每次下单填联系方式即可（匿名下单）？
3. **价格级别数量与命名**：具体几级、每级叫什么（面向谁）。
4. **价格录入方式**：逐档定价 vs. 基于折扣率计算。
5. **订单必填信息**：客户下单时需要哪些字段（姓名、电话、地址/自提、期望时间、备注……）。
6. **产品数据来源**：产品和价格是手工在后台录入，还是需要从 EZ POS 导入/同步？
7. **与 EZ POS 的对接**：初期确认为纯人工；未来是否有 API/导出可对接？
8. **通知机制**：新订单是否需要提醒门店（后台角标、邮件、短信、微信等）？
9. **部署环境**：服务器/域名/HTTPS、数据库托管方式。

## 8. 建议的下一步（开发路线草案）

**里程碑 0 — 立项与设计（当前）**
- 确认第 7 节的关键决策。
- 定稿数据模型与 API 接口约定，写入 `catalog-docs`。

**里程碑 1 — 后端骨架**
- `catalog-service`：建库连库（PostgreSQL）、Product/Price/Order 实体与基础 CRUD API。
- 定义鉴权方案（后台登录）。

**里程碑 2 — 客户端最小可用**
- `catalog-web`：产品列表 + 详情 + 价格展示 + 购物车 + 下单。

**里程碑 3 — 后台最小可用**
- `catalog-backoffice-web`：登录 + 订单列表/详情/改状态 + 产品与价格管理。

**里程碑 4 — 打磨与上线**
- 移动端体验优化、权限完善、部署上线、订单通知。

## 附录：环境与约定

- **后端**：Java 25，`./mvnw spring-boot:run` 启动；包根 `catalog.ezpos.app.catalog`。
- **前端**：pnpm 管理，`pnpm dev` 启动（两个前端需区分端口）。
  - `catalog-web`：Next.js 16 + antd-mobile；SSR 首屏样式按 [antd-mobile SSR 指南](https://mobile.ant.design/guide/ssr/) 配置，可参考 [`antd-mobile-samples/web-ssr`](https://github.com/ant-design/antd-mobile-samples/tree/master/web-ssr)。
  - `catalog-backoffice-web`：改用 Vite + React + antd（需从当前 create-next-app 初始态迁移）。
- **数据库**：PostgreSQL（连接配置待补充到 `application.yaml`）。
- **注意**：`catalog-web` 的 Next.js 版本为 16.x，API 与旧版本有差异，开发前请参考 `node_modules/next/dist/docs/`（见 `AGENTS.md`）。
